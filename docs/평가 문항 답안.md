# mini_redis 평가 문항 답안

이 문서는 `평가 문항.md`의 문항 순서에 맞춰, 현재 `03-1.mini_redis` 구현을 기준으로 한 **모범 답안·설명 초안**이다. 시험·발표에서는 자신의 코드와 용어로 다시 말하면 된다.

---

## 항목 1 — 기능·동작 검증

### String 타입 및 기본 명령

- `**SET` / `GET` / `DEL` / `EXISTS` / `DBSIZE` / `KEYS`**: `Store`가 문자열 키·값을 보관하고, CLI의 `_DISPATCH`에서 각각 `store.set`, `get`, `delete`, `exists`, `dbsize`, `keys`로 연결된다. `GET`은 없으면 `(nil)`, 있으면 `"값"` 형식으로 응답한다.
- **검증**: `tests/test_cli.py`, `tests/test_store.py`에서 위 명령의 응답 형식과 저장소 상태를 함께 검증할 수 있다.

### 메모리·만료·에러

- **LRU 자동 제거**: `maxmemory > 0`이고 `used_memory > maxmemory`이면 `_enforce_maxmemory`가 반복해서 **LRU 리스트의 tail(가장 오래된 키)**에서 키를 꺼내 `HashMap`에서 제거하고 `evicted_keys`를 증가시킨다. 새로 쓰인 키는 `insert_front`로 MRU 쪽에 붙는다.
- `**INFO memory`**: `used_memory`, `maxmemory`, `evicted_keys`를 줄 단위로 출력한다(`cli._fmt_info_memory`). 스냅샷은 `Store.info_memory()`에서 `_expire_due()` 이후 메트릭을 복사해 반환한다.
- `**EXPIRE` / `TTL`**: 존재하는 키에 대해 `expire_at = clock + seconds`를 저장하고 `(expire_at, key)`를 **최소 힙**에 넣는다. `seconds <= 0`이면 즉시 삭제한다. `TTL`은 없으면 `-2`, TTL 없으면 `-1`, 있으면 남은 초를 `ceil(expire_at - now)`로 돌려준다. 모든 공개 연산 앞에서 `_expire_due()`로 만료 시각이 지난 힙 항목을 정리한다(엔트리의 `expire_at`과 힙 꼭대기가 일치할 때만 실제 삭제 — 재설정 등으로 생긴 **stale 힙 레코드**는 건너뜀).
- **에러**: 알 수 없는 명령·인자 개수 오류·정수 파싱 실패는 `CommandError` → `(error) ERR ...` 형식. 단일 엔트리가 `maxmemory`를 초과하는 `SET`은 `OOMError` → `(error) OOM command not allowed when used_memory > 'maxmemory'`로 매핑된다(`cli.dispatch`).

## 항목 2 — 자료구조 설계

### 이중 연결 리스트

- `**Node`**: `prev`, `next`, `data`를 갖고, 리스트는 `_head` / `_tail`로 양끝을 잡는다.
- **O(1) 연산**: `insert_front` / `insert_back`은 새 노드를 머리·꼬리에 붙이기만 하면 된다. `**remove_node(node)`**는 “이미 그 리스트에 속한 노드”를 전제로 이웃 포인터만 바꾸면 되므로 **O(1)**이다. LRU 갱신에 쓰는 **`move_to_front`**는 이미 노드 참조가 있으면 `remove_node` 후 머리에 다시 연결해 **O(1)**이다.

### 해시맵

- **해시 함수**: `_fnv1a32`가 키 문자열을 **UTF-8 바이트**로 인코딩한 뒤, FNV-1a 32비트 누적(`offset basis` XOR 바이트, 곱 prime, `& 0xFFFFFFFF`)으로 해시값을 만든다. 버킷 인덱스는 `해시 % len(_buckets)`이다.
- **충돌 해결**: **분리 연결(separate chaining)** — 개방 주소법과 달리, 충돌한 항목을 **버킷 배열 밖**의 연결 리스트에 쌓는다.
  - **버킷 배열**: `_buckets: list[DoublyLinkedList | None]`. 초기 길이 16, 비어 있으면 `None`, 첫 삽입 시 `DoublyLinkedList()`를 만들어 붙인다.
  - **인덱스**: `_fnv1a32(key) % len(_buckets)`로 슬롯을 고른 뒤, 그 슬롯의 체인만 본다.
  - **버킷 내부 구조(왜 이중 연결 리스트?)**: 체인 한 칸은 `Node`, `data`는 `(key, value)` 튜플. 조회·삽입은 `_find_node`가 head부터 `cur.next`로 키를 비교하는 **선형 탐색**; 새 키는 `insert_front`, 삭제는 `remove_node`로 **O(1)**. 단일 연결 리스트나 배열도 가능하지만, LRU에 이미 쓰는 `DoublyLinkedList`를 재사용하고, 노드 참조가 있을 때 삭제·이동이 O(1)이라는 점에서 통일했다.
  - **개방 주소법과의 차이 (발표용)**:

    |       | 분리 연결(이 구현)                              | 개방 주소법                                          |
    | ----- | ---------------------------------------- | ----------------------------------------------- |
    | 저장 위치 | 버킷은 “체인 헤드”만, 실제 `(k,v)`는 리스트 노드         | 테이블 슬롯 자체에 항목을 둠                                |
    | 충돌 시  | 같은 인덱스의 **별도 체인**에 추가                    | 빈 슬롯이 나올 때까지 **선형·이차·이중 해싱 등으로 probe**          |
    | 로드 팩터 | 엔트리 수 / 버킷 수 > 0.75면 재해시(체인 길이는 1 초과 가능) | 보통 1 미만을 유지해야 함(꽉 차면 삽입 불가)                     |
    | 삭제    | 체인에서 노드만 제거                              | tombstone(삭제 표시) 없으면 probe 경로가 끊길 수 있음          |
    | 군집화   | **버킷 단위 체인**이 길어짐                        | **probe 경로**가 길어지는 primary/secondary clustering |
    | 캐시    | 노드가 흩어져 지역성은 약할 수 있음                     | 연속 배열이라 순회 probe는 캐시에 유리한 경우도 있음                |

  - **한 줄 요약**: “해시로 버킷을 고르고, 그 버킷의 이중 연결 리스트를 따라 같은 키를 찾거나, 없으면 앞에 `(key, value)` 노드를 붙인다.” 개방 주소법은 “같은 배열 안에서 다음 칸을 찾아가는” 방식이고, 체이닝은 “배열 밖 체인에 쌓는” 방식이다.
- **확장**: `_size / len(_buckets) > 0.75`이면 `_resize`에서 기존 `(k, v)` 쌍을 모은 뒤 버킷 길이를 **2배**로 늘리고, `self._size = 0`으로 초기화한 다음 `put`으로 전부 다시 넣어 **전체 재해시**한다.

## 항목 3 — LRU·TTL·통합 흐름

### LRU 아키텍처

- **해시맵**: 키 → `Entry`(값, 바이트 계산, LRU 노드, `expire_at`)를 **상수 시간에 가깝게** 찾기 위함이다.
- **이중 연결 리스트**: “최근 사용 순서”만 표현한다. **키 문자열**이 노드 `data`로 들어가며, MRU는 head, LRU는 tail이다.
- **둘 다 필요한 이유**: 맵만 있으면 임의 키의 LRU 순서를 O(1)로 바꾸기 어렵고, 리스트만 있으면 키 조회가 O(n)이다. **맵으로 노드를 찾고, 리스트로 순서만 O(1) 갱신**하는 조합이 정석이다.

### O(1) LRU (조회 + 갱신)

- **조회**: 해시로 `Entry`와 그 안의 `lru_node`를 얻는다.
- **갱신(히트 시)**: `move_to_front(lru_node)`로 MRU로 옮긴다. 치출은 tail에서 `remove_back`으로 LRU 키를 얻는다.

### TTL·eviction·GET 흐름

- **힙을 쓰는 이유**: 만료 시각 `(expire_at, key)`의 **최솟값**(가장 이른 만료)을 반복해서 꺼내야 하므로 **이진 최소 힙**이 적합하다. 매번 전체를 훑지 않고도 “지금 만료된 것들”을 루트부터 순차적으로 처리할 수 있다.
- **메모리 초과 시 eviction (요약)**  
  1. `_expire_due()`로 먼저 TTL 만료분을 반영(이때 `used_memory`에서 해당 엔트리 바이트 차감).
  2. 여전히 `used_memory > maxmemory`이면 tail에서 LRU 키를 꺼내 엔트리 바이트만큼 `used_memory`에서 빼고 맵에서 제거하며 `**evicted_keys += 1`**.
  3. 조건을 만족할 때까지 반복.
- `**GET` 전체 흐름 (이 구현 기준 순서)**  
  1. `_expire_due()` — 만료된 키 정리(TTL 관점에서 “지금 유효한지”를 맞춤).
  2. 해시맵에서 키로 `Entry` 조회.
  3. 없거나 엔트리가 아니면 `**None` → `(nil)`**.
  4. 있으면 `**move_to_front`**로 LRU 갱신(히트를 MRU로).
  5. `**value` 반환**.

## 항목 4 — 확장·설계 논의

### LFU로 바꾼다면 (4-1)

#### 현재 LRU(단일 리스트) 구조의 한계

지금 `Store`는 `**HashMap`(키→`Entry`) + `DoublyLinkedList` 하나(`_lru`)**로 “최근 사용 순서”만 표현한다.

- 리스트 **head = MRU**, **tail = LRU** — 순서의 의미가 **시간(최근성)** 뿐이다.
- `GET` 히트 시 `move_to_front(lru_node)`로 **“방금 썼다”**만 반영하고, **몇 번 썼는지(빈도)**는 어디에도 없다.
- eviction도 tail에서 꺼내므로 **“가장 오래 안 쓴 키”**를 보낸다. 한 번만 최근에 접근한 키가, 여러 번 쓰인 핵심 키보다 앞에 올 수 있는 등 **LFU가 요구하는 “덜 자주 쓰인 것” 기준과 맞지 않는다**.
- 따라서 LFU로 바꾸려면 **리스트 하나의 전역 순서를 그대로 재해석할 수 없고**, “빈도”라는 **다른 축으로 인덱싱**하는 구조로 바꿔야 한다. TTL용 `MinHeap`은 그대로 둘 수 있지만, **치출 정책용 `_lru`는 LFU 전용 구조로 교체**하는 편이 자연스럽다.

#### 전환 방향: 빈도를 키로 하는 해시(맵) + 빈도별 리스트

LFU에서 필요한 정보는 두 가지다.


| 필요한 질문                   | 역할               | mini_redis 스타일 구현                                                                                       |
| ------------------------ | ---------------- | ------------------------------------------------------------------------------------------------------- |
| 이 키의 빈도는?                | `GET`/`SET` 시 갱신 | `**Entry`에 `freq` 필드** — 별도 `key→freq` 해시맵은 만들지 않고, 지금 `expire_at`·`lru_node`를 두는 것과 같이 **값 객체에 함께 둔다** |
| 빈도 *f*인 키들은? eviction 후보 | 최소 빈도에서 제거       | `**freq_buckets: dict[int, DoublyLinkedList]`** (또는 `HashMap`으로 빈도→리스트) — **빈도를 키로 하는 해시/맵**            |


동작 요약:

1. **저장·조회**: 기존처럼 `_entries`(키→`Entry`)로 O(1)에 가깝게 찾는다. `Entry`에 `freq`(초기 1), 그 빈도 버킷 리스트용 `**lfu_node`**를 둔다.
2. **히트(`GET`)**: `entry.freq`를 1 올린 뒤, **이전 빈도 리스트**에서 `remove_node`, **새 빈도 리스트** head로 `insert_front`. 같은 빈도 안에서는 **리스트 tail = 그 빈도에서 가장 오래 안 쓴 키**(tie-break용 mini-LRU).
3. **eviction**: `**min_freq`**(현재 존재하는 최소 빈도)를 가리키고, `freq_buckets[min_freq]`의 **tail** 키를 제거한다. 해당 버킷이 비면 `min_freq`를 올리거나 다음 비어 있지 않은 빈도를 찾는다.

이렇게 하면 **“빈도 기준 해시 테이블(맵)”**로 치출 후보 집합을 나누고, **키→빈도는 `Entry` 필드**로, **빈도→키 묶음은 맵 + 이중 연결 리스트**로 처리한다. LRU 때와 같이 **노드 참조(`lfu_node`)가 있어야** 빈도 버킷 간 이동이 O(1)에 가깝다.

#### LRU 단일 리스트만으로 LFU를 흉내 내면 안 되는 이유

- 리스트 순서를 “빈도 순”으로 유지하려면 히트마다 **전 리스트에서 자기 위치를 찾아 재삽입**해야 하는데, 빈도는 키마다 달라 **한 줄로 정렬된 순서 하나로 표현하기 어렵다**.
- tail eviction은 여전히 **시간 기준**이라, “자주 쓰이지만 방금 한참 전에 한 번 더 쓴 키”와 “한 번도 안 쓰인 키”를 LFU 정의대로 구분하지 못한다.
- 빈도가 1, 2, …, *k*까지 퍼지면 **빈도별 그룹**이 필요하고, 그게 곧 `**freq → 키 목록` 맵**이다. 이건 LRU의 `_lru` 한 줄로 대체할 수 없다.

#### LRU 대비 트레이드오프 (발표용 한 줄)

- **장점**: 스캔·반복 접근 패턴에서 “자주 쓰는 키”를 더 오래 유지할 수 있다.
- **단점**: 히트마다 **빈도 증가 + 버킷 간 리스트 이동 + `min_freq` 유지**로 `**GET` 경로가 LRU보다 무겁고**, 빈도 상한·aging(오래된 빈도 감쇠) 같은 **추가 정책**을 넣지 않으면 오래된 키가 영원히 안 나갈 수 있다(Redis LFU도 log/increment decay를 둔다).

#### 현재 코드에서 바꿀 지점 (요약)

- `Entry`: `lru_node` → `lfu_node`, `freq` 추가.
- `Store._lru` → `_freq_buckets` + `_min_freq`.
- `_enforce_maxmemory` / `get` 히트 처리: `move_to_front` 대신 **빈도 버킷 갱신** 로직으로 교체.
- `_entries`(`HashMap`)와 TTL `_ttl_heap`은 **키 조회·만료** 용도로 그대로 재사용 가능.

### 데이터가 ~10만 건일 때 병목·개선

- **체이닝**: 해시가 좋아도 최악·편향 시 **특정 버킷 체인 길이**가 길어지면 `get`/`put`이 선형에 가까워질 수 있다 → **버킷 수·해시 품질**, 필요 시 **다른 해시/이중 해싱** 등 검토.
- `**KEYS`**: 전체 키를 돌아 **O(n)** — 프로덕션 Redis도 위험하다고 알려진 패턴과 동일. 현재 구현은 `HashMap.keys()`로 **모든 버킷·체인을 끝까지 순회**한 뒤 `list(...)`로 한 번에 모으므로, 키가 10만 건이면 **한 번의 `KEYS`가 그만큼 전부 스캔 + 문자열 리스트 할당**까지 한꺼번에 일어난다. 다른 `GET`/`SET`이 길게 막히거나 응답이 커지는 **지연·메모리 스파이크**가 난다.
  - `**SCAN` 스타일 순회(점진적 커서)**: Redis의 `SCAN cursor [MATCH pattern] [COUNT hint]`처럼, **한 번에 전부가 아니라 조금씩** 돌려준다. 클라이언트는 `cursor=0`으로 시작해, 서버가 `(다음 cursor, 이번에 걸린 키 일부)`를 반환하고, **cursor가 다시 0이 될 때까지** 반복한다. 구현 아이디어는 (1) 해시 테이블 **버킷 인덱스 + 체인 내 위치**를 cursor 상태로 저장하고, (2) 한 호출당 최대 `COUNT`개만 yield, (3) 재호출 시 그 위치부터 이어가기다. 전체 키 수는 여전히 O(n)이지만 **작업을 여러 번 나누므로** 한 요청의 최악 지연을 줄이고, 중간에 다른 명령을 끼워 넣기 쉽다(완전한 “논블로킹”은 아니지만 *운영에서 `KEYS`  한 방보다 안전**하다고 설명할 수 있다).
  - **인덱스·조회 제한**: “전 키 나열” 자체를 줄이는 쪽이다. 예: `**MATCH` 패턴**(glob)으로 후보를 좁히기, `**COUNT` 상한*으로 한 번에 돌려주는 개수 제한, 또는 **별도 보조 인덱스**(접두사 트리, 태그→키 집합)를 두어 `user:` 같은 질의만 O(해당 집합 크기)로 처리하기. mini_redis 수준에서는 최소한 `**KEYS`에 패턴·limit 인자를 두지 않는 한** 10만 건 DB에서 `KEYS`는 피하고 `DBSIZE`·단건 `GET`·(있다면) `SCAN`을 쓰는 것이 맞다.
- **재해시**: 로드 팩터 초과 시 **한 번에 전 테이블 재삽입**은 순간 지연(latency spike)이 크다 → 점진적 재해시(incremental rehashing) 이야기 가능.
- **힙 stale 항목**: TTL 재설정 시 힙에 남는 오래된 `(expire_at, key)`는 `pop` 시 `Entry.expire_at`과 비교해 무시한다. 키가 매우 많고 EXPIRE가 잦으면 **힙 크기 대비 실제 만료 처리 비용**이 커질 수 있어, 주기적 압축·lazy delete 정책을 논의할 수 있다.
- **Python 객체 오버헤드**: 교육용 구현은 `Entry`, `Node`, `str` 등 **런타임 객체 비용**이 크다 — 순수 C/고정 레이아웃 대비 **메모리 한도 대비 실제 저장 가능 키 수**가 달라진다는 점을 병목으로 말할 수 있다.

### `used_memory`에 자료구조 오버헤드를 넣으면

- **달라지는 점**: 지금은 대략 `**len(key UTF-8) + len(value UTF-8)`**만 누적한다(`Entry.entry_bytes`). 노드·해시 버킷·힙 배열·파이썬 객체 헤더까지 넣으면 **같은 키/값이라도 계정이 커지고**, 같은 `maxmemory`에서 **더 일찍 eviction**이 나거나 단일 `SET`이 OOM에 더 쉽게 걸릴 수 있다.
- **공정한 비교·채점 보정**: “무엇을 1바이트로 칠지”를 **명세로 고정**(예: 키+값만 / + LRU 노드 포인터 근사 / + 해시 테이블 슬롯당 분담 등). 구현체마다 오버헤드 모델이 다르면 **같은 시나리오라도 치출 시점이 달라지므로**, 벤치마크는 **동일 계정 규칙** 또는 **상대 비교(비율)**로 맞추는 것이 타당하다고 설명하면 된다.
