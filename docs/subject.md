# Mini Redis 미션

---

## 1. 미션 소개

Redis를 써봤는데 왜 이렇게 빠른지 설명해보라고 하면 막히는 분들이 많습니다. 이유가 있습니다. **내부 자료구조를 직접 구현해본 사람이 드물기 때문**입니다. 해시맵, 이중 연결 리스트, 힙을 밑바닥부터 짜면서 **LRU**와 **TTL**이 어떻게 동작하는지 손으로 확인합니다.

Redis는 전 세계에서 가장 널리 사용되는 **In-Memory Key-Value** 데이터 저장소입니다. 캐시, 메시지 브로커, 세션 저장소 등 다양한 용도로 활용되며, 그 핵심에는 효율적인 자료구조가 있습니다.

이번 미션에서는 Redis의 핵심 기능을 직접 구현하며 **CLI 기반 Mini Redis**를 완성합니다. 해시맵, 이중 연결 리스트, 힙 같은 자료구조를 밑바닥부터 구현하면서 평소 당연하게 사용하던 내장 자료형의 내부 동작 원리를 깊이 이해하게 됩니다.

또한 실제 Redis가 메모리 제한 환경에서 **LRU**로 오래된 데이터를 자동 제거하고, **TTL**로 만료 시간을 관리하는지 직접 구현하며 체득합니다. 이 경험은 이후 알고리즘 학습, 코딩 테스트, 실제 서비스 성능 최적화로 확장될 수 있습니다.

## 2. 최종 결과물

다음 기능이 정상 동작하는 **CLI 기반 Mini Redis 프로그램 1개**를 완성한다.

### 2.1 String 타입 기본 명령어(6개)

| 명령 | 동작 |
|------|------|
| **SET** | 키에 값 저장. 성공 시 내부적으로 **LRU 추적 업데이트** |
| **GET** | 키의 값 조회. 성공 시 내부적으로 **LRU 추적 업데이트** |
| **DEL** | 키 삭제 |
| **EXISTS** | 키 존재 여부 확인 |
| **DBSIZE** | 전체 키 개수 반환 |
| **KEYS** | 전체 키 목록 출력(**패턴 매칭은 구현하지 않음**) |

### 2.2 메모리 관리 명령어(2개)

| 명령 | 동작 |
|------|------|
| **CONFIG SET maxmemory** | 최대 메모리 제한 설정(**바이트** 단위) |
| **INFO memory** | 현재 메모리 사용량, 제한, 제거된 키 개수 확인 |

### 2.3 TTL 관리 명령어(2개)

| 명령 | 동작 |
|------|------|
| **EXPIRE** | 키의 만료 시간 설정(**초** 단위) |
| **TTL** | 키의 남은 만료 시간 조회 |

### 2.4 CLI 인터페이스

- 사용자가 명령을 입력하면 즉시 실행 결과를 확인할 수 있는 **REPL** 환경
- 명령어 파싱 → 실행 → 결과 출력이 반복적으로 동작

## 3. 과제 목표

이 과제를 마친 후, 학습자는 아래를 스스로 설명할 수 있어야 한다.

1. 해시맵의 **해시 함수**와 **충돌 해결 방식(체이닝)**을 구현 코드를 기반으로 설명할 수 있다.
2. **이중 연결 리스트**와 **해시맵**을 조합하여 **O(1) LRU 추적**이 가능한 이유를 설명할 수 있다.
3. **힙**이 TTL 만료 시간 관리에 적합한 이유를 설명할 수 있다.
4. 메모리 제한 환경에서 **LRU 정책**으로 데이터를 제거하는 전체 흐름(`used_memory` 갱신 포함)을 설명할 수 있다.

## 4. 기능 요구 사항

다음 요구사항을 **모두** 만족해야 한다.

### 4.1 기본 자료구조 직접 구현

**내장 Key-Value 컬렉션으로 대체 금지.**

#### 이중 연결 리스트

| 항목 | 요구 |
|------|------|
| **노드** | `prev`, `next`, `data` 필드 |
| **메서드** | `insert_front`, `insert_back`, `remove_front`, `remove_back`, `remove_node`, `move_to_front` |
| **복잡도** | 모든 삽입·삭제·이동 연산 **O(1)** |

#### 해시맵(체이닝)

| 항목 | 요구 |
|------|------|
| **메서드** | `put`, `get`, `remove`, `contains`, `keys`, `size` |
| **해시 함수** | 직접 설계 |
| **충돌** | **체이닝**(권장: 이중 연결 리스트 재사용) |
| **확장** | 로드 팩터 **0.75 초과** 시 버킷 **2배** 확장 |

#### 힙(최소 힙)

| 항목 | 요구 |
|------|------|
| **메서드** | `push`, `pop`, `peek`, `size` |
| **내부** | `_heapify_up`, `_heapify_down` 구현 |
| **TTL** | `(expire_at, key)` 형태 요소 다루기 |

### 4.2 String 타입 명령어(Redis 스타일 출력)

#### 공통 규칙

- 키 기반 명령은 실행 전 **만료 여부**를 먼저 확인(만료된 키는 삭제 후 ‘없는 키’처럼 처리).
- 출력은 Redis 스타일: `OK`, `(nil)`, `(integer) N`, `(error)` …

| 명령 | 동작·출력 |
|------|-----------|
| **SET key value** | 성공 시 `OK`. 메모리 초과 시 **LRU 제거**(세부 규칙은 §4.4). 기존 키 덮어쓰기 시 **기존 TTL 초기화(삭제)** |
| **GET key** | 없거나 만료 시 `(nil)`. 존재 시 `"value"` 형태. **반환 성공 시에만** LRU 갱신(만료로 삭제된 경우 갱신 없음) |
| **DEL key** | 성공 `(integer) 1`, 없으면 `(integer) 0`. LRU/TTL 구조에서도 함께 제거 |
| **EXISTS key** | 존재 `(integer) 1`, 없으면 `(integer) 0` |
| **DBSIZE** | `(integer) N` |
| **KEYS** | 전체 키 배열 형태(정렬·순서 비요구). 예: `"user:2"`, `"user:3"`. 없으면 `(empty array)` 등으로 비어 있음 표현 가능 |

### 4.3 메모리 관리 + LRU 자동 제거

#### CONFIG SET maxmemory bytes

- `bytes`: **0 이상** 정수. **0**은 **무제한**.
- 성공 `OK`, 정수 파싱 실패 시 에러 표준 준수.

#### INFO memory

아래 **3개 항목** 최소 포함(표현 형태는 동일하면 됨).

- `used_memory:<number> maxmemory:<number> evicted_keys:<number>`

#### used_memory 산정(공식)

`used_memory = Σ( len(utf8(key)) + len(utf8(value)) )`

- 자료구조(노드·포인터·버킷 등) **오버헤드는 계산에서 제외**.

#### LRU 제거 규칙

- `maxmemory > 0`이고, **SET 이후** `used_memory`가 `maxmemory`를 초과하면, `used_memory`가 `maxmemory` 이하가 될 때까지 **가장 오래 사용되지 않은 키(LRU)**부터 제거.
- 제거된 키는 `evicted_keys`에 **누적 카운트**.
- **단일 엔트리(키+값)**가 `maxmemory`를 초과하면: 저장하지 않고 **에러(OOM)** 출력.

### 4.4 TTL 관리(힙 기반)

| 명령 | 동작 |
|------|------|
| **EXPIRE key seconds** | 키 없으면 `(integer) 0`. `seconds` **0 이하**면 즉시 만료 처리 가능(존재하면 삭제 후 `(integer) 1`). 정상 `(integer) 1` |
| **TTL key** | 없으면 `(integer) -2`. 만료 시간 없으면 `(integer) -1`. 있으면 남은 초 `(integer) N` |

#### TTL/LRU 엣지 케이스(명시)

- 만료된 키는 **GET** 시 먼저 삭제 후 `(nil)`, **LRU 갱신 없음**.
- **SET**이 기존 키 덮어쓸 때 **TTL 초기화(삭제)**.
- **EXPIRE**를 없는 키에 호출하면 `(integer) 0`.
- **DEL**은 데이터·TTL·LRU **모든 구조**에서 엔트리 제거.
- 구현: **힙으로 가장 빠른 만료를 빠르게 찾을 수 있어야 함**(lazy deletion 등은 구현 선택).

### 4.5 에러 처리 표준 + CLI

#### CLI

- 프롬프트: `mini-redis>`
- 사용자 입력 읽기 → 파싱 → 실행
- `exit` 또는 `quit`로 종료

#### 에러 출력(예시)

| 상황 | 형식 |
|------|------|
| 잘못된 명령 | `(error) ERR unknown command '<cmd>'` |
| 인자 개수 오류 | `(error) ERR wrong number of arguments for '<cmd>' command` |
| 정수 파싱 실패 | `(error) ERR value is not an integer or out of range` |
| 메모리 초과(OOM) | `(error) OOM command not allowed when used_memory > 'maxmemory'` |

#### 값 파싱

- `"Alice"` 같은 **따옴표** 입력 허용(난이도에 따라 단순 규칙 가능).
- 최소: **공백 없는 값** 또는 **큰따옴표로 감싼 값** 중 하나 방식 지원.

## 5. 보너스 과제 (선택)

| 항목 | 내용 |
|------|------|
| **동적 배열** | 직접 구현: `append`/`get`/`set`/`remove`, capacity **2배** 확장. 해시맵 버킷·힙 저장소 등에 적용 가능 |
| **스택/큐/덱** | 개념 조사 후 `STACK_QUEUE_DEQUE.md` 문서화. Pub/Sub·커맨드 히스토리 등 큐 기반 기능의 기반 |
| **이진 트리·순회** | 구현 및 전위·중위·후위·레벨 순회. “힙=완전 이진 트리 배열 표현” 관점 강화 |
| **이진 탐색 트리(BST)** | 삽입·탐색·삭제·중위 순회 정렬. 키 정렬·범위 조회 확장의 기반 |
| **Pub/Sub** | `PUBLISH`, `SUBSCRIBE`, 채널 기반 메시징. 연결 리스트를 메시지 큐 등으로 재활용 가능 |
