# 해시맵에 FNV-1a 32-bit를 둔 이유

이 문서는 **Mini Redis**의 `HashMap`에서 문자열 키를 버킷 인덱스로 바꿀 때 **FNV-1a 32-bit**를 직접 구현한 선택을, **다른 대안과의 장단** 관점에서 정리한다. 구현은 `mini_redis/hashmap.py`의 `_fnv1a32`를 기준으로 한다.

## 이 프로젝트에서 해시가 맡는 역할

- 키는 **문자열**이고, UTF-8 바이트열에 대해 **32비트 정수** 하나를 만든 뒤 `버킷 수`로 나머지 연산해 **버킷 인덱스**를 얻는다.
- 충돌은 **별도 체이닝(이중 연결 리스트)**으로 처리하고, **로드 팩터**가 임계값을 넘으면 버킷을 늘리며 **rehash**한다.

따라서 필요한 성질은 대략 다음과 같다.

1. **구현이 단순하고 짧다** — 과제·학습 범위에서 “해시 함수 자체”가 주제가 아니라도, **직접 구현** 요구를 만족해야 한다.
2. **일반적인 문자열 키에 대해 버킷 분포가 지나치게 치우치지 않는다** — 평균적으로 체인 길이가 짧게 유지되면 된다. 암호학적 난수성이나 공격자에 대한 강인성은 **이 CLI 미니 구현의 범위 밖**이다.
3. **표준 라이브러리·외부 패키지에 의존하지 않는다** — `plan.md`에서도 “외부 라이브러리 무의존”을 명시한다.
4. **결정적(deterministic)** — 같은 키는 항상 같은 해시값을 내면, 테스트·재현이 쉽다.

이 네 가지를 만족하는 후보들 사이에서 FNV-1a가 **균형이 좋은 타협점**에 있다고 본다.

## 후보 알고리즘과 장단

### FNV-1a 32-bit (채택)

- **아이디어**: 고정 **오프셋 베이스**와 **소수 곱**을 쓰는 간단한 비암호 해시. **1a** 변형은 바이트마다 **XOR 후 곱하기** 순서(FNV-1은 곱한 뒤 XOR)로, 짧은 키·특정 패턴에서 **분포·충돌 특성이 조금 나은 경우**가 많다고 알려져 있다.
- **장점**
  - 코드가 매우 짧고, 상수(`2166136261`, `16777619`)가 **문헌·구현 예제와 잘 맞아** 검증·비교가 쉽다.
  - 바이트 단위 루프와 비트 마스크(`& 0xFFFFFFFF`)만으로 끝나 **순수 Python에서도 부담이 적다**.
  - “비암호 일반 해시”로서 **해시 테이블 버킷링** 용도에 흔히 쓰인다.
- **단점**
  - **암호학적으로 안전하지 않다** — 키를 조작해 의도적으로 같은 버킷으로 몰아넣는 **Hash-flooding**류 공격에 취약할 수 있다. 다만 **신뢰할 수 없는 클라이언트가 임의 키를 대량으로 넣는 네트워크 서비스**가 아니라, 교육용 CLI 저장소라면 보통 설계 범위 밖이다.
  - Murmur·xxHash 등 최신 비암호 해시 대비 **품질(확산·아발란치)**은 보수적으로 본다. 버킷 수가 적고 rehash 전략이 없다면 차이가 드러날 수 있으나, **동적 rehash + 체이닝**이 있으면 실무 감각상 충분한 경우가 많다.

### FNV-1 (비-1a)

- 구현 난이도·속도는 거의 동일하다.
- **1a**가 XOR 순서만 바꾼 형태라, **문자열 해시 테이블** 맥락에서는 1a를 더 흔히 권하는 편이라 **1a를 택한 것**이 자연스럽다.

### djb2 등 단순 다항식 해시

- **장점**: 곱셈 한 번·덧셈 위주로 **더 짧은 루프**를 만들 수 있다.
- **단점**: 상수 선택과 키 패턴에 따라 **특정 입력에서 버킷 편향**이 나기 쉽다는 이야기가 간혹 있다(엄밀한 비교는 입력 분포 의존). FNV 계열은 **표준화된 파라미터**가 있어 설명·재현이 수월하다.

### Python 내장 `hash(str)`

- **장점**: 구현 비용 제로, C 레벨로 빠를 수 있다.
- **단점**
  - **프로세스마다 salt가 붙는 등** 버전·실행마다 값이 달라질 수 있어, **문서·테스트·다른 언어 포팅**과 “이 키의 버킷 인덱스”를 고정적으로 말하기 어렵다.
  - “직접 설계·구현한 해시 함수” 요구와도 맞지 않는다.

### MurmurHash, xxHash, CityHash 등

- **장점**: **비암호 해시로서의 분포·속도**가 뛰어나고, 대규모 데이터·파이프라인에서 선호된다.
- **단점**: 32비트·간단 루프만으로 끝나지 않고 **라운드·블록 처리**가 들어가 코드량이 커진다. 이 미션에서는 **이득 대비 구현·검증 부담**이 커진다.

### SipHash 등 (HashDoS 대응 계열)

- **장점**: **적대적 입력**에도 버킷이 안정적으로 퍼지도록 설계된 경우가 많다.
- **단점**: **비밀 키**를 두고 초기화해야 하고, 구현 복잡도가 FNV 대비 한 단계 위다. Mini Redis 수준에서는 **과한 방어선**이다.

### 암호학적 해시 (SHA-256 등)

- **장점**: 강한 무작위성에 가까운 출력.
- **단점**: 키 하나마다 **비용이 과하고**, 해시 테이블 버킷용으로는 **명백한 과잉**이다.

## 선택 요약

| 기준            | FNV-1a 32-bit | 내장 `hash` | Murmur/xx 등 | SipHash   |
|-----------------|---------------|-------------|----------------|-----------|
| 구현 단순성     | 매우 좋음     | 해당 없음   | 보통           | 낮음      |
| 외부 의존 없음  | 예            | 예          | 보통(직접 구현 시 예) | 예(직접 구현 시) |
| 결정적·설명 용이 | 예            | 비권장      | 예             | 키 관리 필요 |
| 버킷 분포(일반 키) | 충분한 편   | 좋을 수 있음 | 매우 좋음      | 매우 좋음 |
| 적대적 입력 방어 | 없음          | 환경 의존   | 보통 없음      | 있음      |

**정리**: 이 저장소는 **신뢰된 로컬 CLI**, **체이닝 + 로드 팩터 기반 확장**이 있고, 과제는 **해시를 스스로 구현**하는 것이므로, **짧은 코드·표준 상수·무의존·결정적**을 만족하면서 테이블 용도에 널리 쓰이는 **FNV-1a 32-bit**가 합리적인 기본 선택이다. 네트워크 멀티테넌트 KV처럼 **비신뢰 키 스트림**이 전제라면 SipHash나 랜덤 시드 해시 등 **다른 축**의 논의가 필요하다.

## 구현과의 대응

- `_fnv1a32`는 키를 `utf-8`로 인코딩한 뒤 바이트마다 XOR·곱하기를 반복하고, 마지막에 `& 0xFFFFFFFF`로 **부호 없는 32비트** 범위로 맞춘다.
- 최종 버킷은 `_fnv1a32(key) % len(self._buckets)`로 정해지며, **충돌·성능**은 해시 품질만이 아니라 **버킷 수·rehash 정책**과 함께 설계된다(`docs/plan.md`의 해시맵 절 참고).
