# 인덱싱 공식이 있으면 일차원 배열로 눌러 효율적 접근하기

이 문서는 **완전 이진 트리**나 **이차원 격자**처럼 논리적으로는 계층·격자 구조인 자료를 **연속 메모리(일차원 배열)**에 올렸을 때, **동일한 논리 위치에 O(1)로 도달할 수 있는 조건**을 정리한다.

## 핵심 요약

어떤 자료구조든 **“논리적 주소 → 물리적 인덱스”를 닫힌 형태의 계산으로 표현**할 수 있다면, 굳이 포인터·중첩 리스트·객체 그래프로 표현하지 않고 **일차원 배열 한 덩어리**로 눌러도 된다. 그 결과는 대개 다음과 같다.

- **캐시 친화적**: 인접 논리 요소가 메모리에서도 가깝게 배치되기 쉽다.
- **할당·순회 단순화**: 한 번의 연속 블록으로 크기를 정하고 인덱스만으로 접근한다.
- **접근 비용**: 인덱스 계산이 O(1)이면 **임의 위치 읽기/쓰기도 O(1)**에 가깝다(계산 외 오버헤드는 환경에 따름).

반대로, **인덱스 공식이 없거나 매우 복잡**해지면(희소 구조, 동적 삽입으로 구멍이 많이 생김 등) 일차원 압축은 이득이 줄고, 다른 표현이 더 낫다.

## 완전 이진 트리를 배열로 누르는 경우

0-based 인덱스에서 노드 `i`에 대해 흔히 쓰는 매핑은 다음과 같다.

- **부모**: `(i - 1) // 2` (루트 `i = 0` 제외)
- **왼쪽 자식**: `2 * i + 1`
- **오른쪽 자식**: `2 * i + 2`

**완전 이진 트리**이면 높이 `h`일 때 노드 수가 정해져 있고, **빈 슬롯 없이** `0 .. n-1`까지 연속으로 쓸 수 있어 배열 길이가 `n`으로 딱 맞는다.  
**완전이 아니면** 같은 논리를 쓰더라도 중간 “구멍”을 배열로 표현해야 해서 메모리 낭비나 다른 인코딩(힙은 보통 완전/준완전 가정)이 필요해진다.

즉, “트리”라는 **추상 타입**을 **배열 + 인덱스 규칙**으로 구체화한 것이고, **규칙이 유지되는 한** 포인터 없이도 부모·자식 이동이 가능하다.

## 이차원 격자를 일차원으로 누르는 경우

행 `r`, 열 `c`, 열 개수 `cols`일 때 **행 우선(row-major)** 예시는 다음과 같다.

- **1D 인덱스**: `i = r * cols + c`
- **역변환**: `r = i // cols`, `c = i % cols`

고정된 `rows × cols` 직사각 격자에서는 이 매핑이 항상 성립한다.  
가변 길이 행(래그드 배열)이면 **고정 `cols` 가정이 깨지므로** 단일 공식으로 누르기 어렵고, **누적 오프셋 배열(prefix sum)** 같은 보조 배열을 두는 식으로 “인덱싱”을 재정의해야 한다.

## 공통 패턴으로 읽기

| 표현 | 논리 주소 | 1D 인덱스로 가는 열쇠 |
|------|-----------|------------------------|
| 완전 이진 트리(배열 힙) | 노드 번호 `i` | 자식·부모 식 |
| 고정 격자 | `(r, c)` | `r * cols + c` (또는 열 우선 규칙) |
| 일반 그래프 | 정점 id | **인접 리스트 오프셋** 등 별도 인덱싱 설계 필요 |

정리하면, **“눌렀을 때의 물리 인덱스 = f(논리 주소)”가 명확한가**가 일차원화가 가치 있는지의 분기점이다. `f`가 단순·O(1)이면 **연속 버퍼 + 정수 인덱스**만으로도 충분히 “효율적인 접근”에 가깝게 갈 수 있다.

## 한계와 주의

- **삽입·삭제로 논리 구조가 자주 바뀌면** 배열 압축 표현은 **원소 이동·재배열** 비용이 커질 수 있다. (힙의 `sift` 등이 대표적)
- **희소(sparse)** 데이터는 1D로 펼치면 대부분이 빈 칸이 되어 **밀도가 떨어진다**. 이때는 해시·CSR 같은 **다른 인덱싱**이 나을 수 있다.

이 문서의 메시지는 “무조건 1D가 최선”이 아니라, **인덱싱 계산이 안정적으로 정의될 때 1D 압축이 강력한 도구가 된다**는 쪽에 가깝다.
