# mini_redis 평가 문항

아래 항목별로 **동작 검증**과 **설명·논의**를 구분해 두었습니다. 각 블록 끝의 `PASS`는 해당 항목 통과 기준입니다.

---

## 항목 1 — 기능·동작 검증

### String 타입 및 기본 명령

- `SET`, `GET`, `DEL`, `EXISTS`, `DBSIZE`, `KEYS`가 모두 정상 동작하는가?

### 메모리·만료·에러

- **LRU 자동 제거**: `maxmemory` 설정 후 제한 초과 시 가장 오래된 키가 자동 제거되는가?
- **메모리 정보**: `INFO memory`에서 `used_memory` / `maxmemory` / `evicted_keys`가 규칙에 맞게 출력되는가?
- **TTL 관리**: `EXPIRE` / TTL 규칙이 동작하며, 만료된 키가 적절히 제거되는가?
- **에러 처리**: 잘못된 명령·인자·정수 오류·OOM이 표준 형식으로 출력되는가?

---

## 항목 2 — 자료구조 설계

### 이중 연결 리스트

- 노드 구조(`prev`, `next`, `data`)와 핵심 메서드가 **O(1)**로 동작하도록 어떻게 구성했는지 설명할 수 있는가?

### 해시맵

- 직접 설계한 **해시 함수**가 어떤 입력을 받아 어떤 과정을 거쳐 인덱스를 만드는지 설명할 수 있는가?
- **충돌 해결**을 체이닝으로 어떻게 구현했는지(버킷 내부 구조 선택 포함) 설명할 수 있는가?
- **로드 팩터 0.75 초과** 시 버킷 **2배 확장**을 어떤 절차로 수행하는지 설명할 수 있는가?

---

## 항목 3 — LRU·TTL·통합 흐름

### LRU 아키텍처

- **해시맵 + 이중 연결 리스트**가 각각 어떤 역할을 하고, 왜 둘 다 필요한지 설명할 수 있는가?
- **O(1) LRU** 달성 원리를 **조회(해시) + 갱신(리스트 이동)** 관점에서 설명할 수 있는가?

### TTL·eviction·GET 흐름

- TTL 관리에 **힙(heap)**을 사용한 이유(가장 빨리 만료되는 시점을 빠르게 찾는 성질)를 설명할 수 있는가?
- 메모리 초과 시 **eviction** 흐름을 단계별로 설명할 수 있는가? (`used_memory` 산정·갱신, LRU 제거, `evicted_keys` 증가 포함)
- **GET** 명령 전체 흐름을 순서대로 설명할 수 있는가? (TTL 확인 → 삭제 여부 → 값 반환 → LRU 갱신 조건)

---

## 항목 4 — 확장·설계 논의

- **LFU 정책**: LRU 대신 LFU를 구현한다면 자료구조를 어떻게 바꿔야 하는가? 합리적으로 답할 수 있는가?
- **규모·병목**: 데이터가 약 10만 건으로 늘어날 때 현재 구조에서 병목이 될 수 있는 부분과 개선 방안을 말할 수 있는가?
- `**used_memory` 모델**: 자료구조 오버헤드까지 포함하는 모델로 바꾸면 무엇이 달라지고, 공정한 비교·채점을 위해 어떤 보정이 필요한지 설명할 수 있는가?
